## Project Resource Address Change

The internal address of the project resource changed from `mongodbatlas_project.this` to `mongodbatlas_project.this[0]`.

### Why the change?

The project resource now uses `count` to support reference mode (`project_id`), which changes its Terraform address.

### Who is affected?

All existing users. When upgrading, `terraform plan` will show a move operation:

```text
mongodbatlas_project.this has moved to mongodbatlas_project.this[0]
```

### Required action

No action required. The `moved` block included in the module handles this automatically. The move is safe and does not recreate the resource.

## Tags Default Value Change (Breaking Change)

The default value of `tags` changed from `{}` to `null`.

### Why the change?

The project resource defaults to `null` when no tags are set on a project. The previous `{}` default caused a plan diff when importing a project with no tags.

### Who is affected?

Users who did not set `tags` explicitly. After upgrading, `terraform plan` will show an in-place update changing `tags` from `{}` to `null`.

### Required action

No action required. The change is safe. If you want a zero-diff upgrade, set `tags = {}` explicitly before upgrading.

## Default Alert Settings Now Off by Default (Breaking Change)

The default value of `with_default_alerts_settings` (create-only attribute) changed from `true` to `false`.

### Why the change?

Customer feedback indicated that the default Atlas alert settings often require immediate customization.

### Who is affected?

Users who did not set `with_default_alerts_settings` explicitly and whose project was created with the old default (`true`) will see an error:
```text
│ Error: with_default_alerts_settings cannot be updated or set after import, remove it from the configuration or use the state value (see below).
```

### Required action

Set `with_default_alerts_settings = true` explicitly when upgrading to prevent the error:

```hcl
module "atlas_project" {
  # ...
  with_default_alerts_settings = true
}
```

## IP Access List Allow-All Validation (Breaking Change)

The module now rejects "allow-all" IP access list entries: `0.0.0.0/0` & `::/0`.

### Why the change?

Allow-all entries are overly permissive. The module now blocks these entries by default to prevent accidental exposure.

### Who is affected?

Users with `0.0.0.0/0` or `::/0` in their `ip_access_list` will get a validation error.

### Required action

If the allow-all entry is intentional, set `skip_allow_all_validation = true` on the entry. Otherwise, remove the allow-all entry from your configuration.
